# 管理侧用户、广告接入、收益报表与路由修复设计

## 1. 背景与目标

本次改造解决四类相互关联的问题：

1. 当前左侧【用户管理】实际承载的是代理商/租户管理，名称与内容不一致，也没有真正的 App 用户管理入口。
2. 超级管理员和代理商管理员的数据范围必须由服务端强制隔离，且代理商不是固定枚举，所有筛选项均来自实时代理商数据。
3. 当前【广告接入】混入未参与运行的字段、收益报表字段和只读系统状态，造成配置成本高、必填语义不清。
4. 广告中心子菜单生成了 `/skit/ad-center/skit/ad-consumption` 和 `/skit/ad-center/skit/ad-record`，Vue 客户端路由无法匹配并显示 404。

目标是形成清晰、可验证的管理侧信息架构，并在不破坏现有有效配置和历史链接的前提下修复路由。

## 2. 方案选择

### 2.1 用户管理信息架构

采用左侧父子菜单，而不是在同一页面使用页签：

- 【用户管理】作为父菜单。
- 【代理商管理】承载现有代理商/租户工作台。
- 【用户管理】展示真实 App 用户。

这样可直接通过 URL 定位页面，权限、面包屑和菜单高亮也能独立控制。

### 2.2 广告与收益配置拆分

采用同级功能标题，不把收益配置继续嵌在广告接入长表单中：

- 【广告接入】：仅保留会影响 SDK 初始化、短剧目录同步、广告请求和激励广告位的运行参数。
- 【收益报表】：承载 Taku 官方收益报表凭证、时区、币种、金额精度、同步状态和权限诊断。

不采用“保留全部字段但标成可选”的方案，因为无运行作用的字段仍会误导运营人员。

### 2.3 路由修复

采用修复公共路径解析器并增加旧链接兼容跳转的组合方案：

- 公共解析器遵守 Vue Router 语义：以 `/` 开头的绝对子路由直接返回，不再拼接父路径。
- 相对子路由继续基于父路径拼接。
- 对已经外泄或收藏的两个重复路径增加隐藏兼容跳转。

不在单个菜单组件中写特例，因为同一个解析器还被菜单、面包屑、标签页和 TabMenu 共用。

## 3. 菜单与路由设计

### 3.1 用户管理

推荐路由：

| 菜单 | 路由 | 页面 |
| --- | --- | --- |
| 用户管理（父级） | `/skit/user-center` | 仅作为路由容器 |
| 代理商管理 | `/skit/user-center/agents` | 复用并整理现有租户工作台 |
| 用户管理 | `/skit/user-center/users` | 新的 App 用户列表 |

兼容现有 `/skit/user`：跳转到 `/skit/user-center/agents`，避免旧书签失效。

### 3.2 广告中心

正式路由保持为：

- 消费明细：`/skit/ad-consumption`
- 广告监控：`/skit/ad-record`

错误路径兼容：

- `/skit/ad-center/skit/ad-consumption` → `/skit/ad-consumption`
- `/skit/ad-center/skit/ad-record` → `/skit/ad-record`

兼容路由必须隐藏，不出现在菜单、面包屑或权限配置界面中。

### 3.3 路径解析规则

`pathResolve(parentPath, path)` 的契约：

| 输入 | 输出 |
| --- | --- |
| `('/skit/ad-center', '/skit/ad-record')` | `/skit/ad-record` |
| `('/skit/ad-center', 'ad-record')` | `/skit/ad-center/ad-record` |
| `('/skit/ad-center', '')` | `/skit/ad-center` |
| 任意父路径 + 外部 URL | 原外部 URL |

## 4. 权限与租户隔离

### 4.1 超级管理员

- 可查看全平台代理商和 App 用户。
- 代理商筛选项通过服务端实时返回，不在前端固定编码。
- 用户列表显示所属代理商/租户。
- 所有跨租户查询都必须经过超级管理员角色校验。

### 4.2 代理商管理员

- 服务端从登录身份解析租户，不接受前端传入的任意 tenantId 扩大范围。
- 用户列表、广告消费、广告监控、收益报表和配置读取均自动限定当前租户。
- 页面不显示代理商筛选器，也不允许通过改 URL、查询参数或请求体访问其他租户。

### 4.3 动态代理商

- 代理商筛选、代理商名称和租户状态均来自代理商接口实时数据。
- 新增、停用、改名后无需发布前端即可生效。
- 历史记录保留代理商标识；代理商停用后仍可按权限查看其历史数据。

## 5. 代理商管理与 App 用户管理

### 5.1 代理商管理

现有 `/skit/user` 的租户工作台迁移到【代理商管理】，保留：

- 代理商新增、编辑、启停和审计。
- 广告与收益配置入口。
- 代理关系、分成和结算相关功能。

只有超级管理员可以手动新增代理商；代理商管理员不得获得平台级新增能力。

### 5.2 App 用户管理

用户列表至少展示：

- 用户 ID、手机号/账号脱敏值、昵称、状态。
- 所属代理商/租户（仅超级管理员展示）。
- 注册来源、注册时间、最近登录时间。
- 可用积分或业务账户摘要（以服务端已有字段为准，不用前端虚构）。

查询条件至少包括账号/手机号、状态、注册时间；超级管理员额外支持动态代理商筛选。

所有修改类操作都需要明确权限、二次确认和操作审计；本期若服务端没有可靠的用户修改能力，则先交付只读列表与详情，不使用前端假数据补齐。

## 6. 广告接入

### 6.1 保留字段

穿山甲目录同步：

- 是否启用。
- App ID。
- Server Key / App Secret。

Taku 激励广告运行：

- 是否启用。
- Taku App ID。
- Taku App Key。
- 激励视频 Placement ID。

运行状态以独立只读状态卡展示，而不是伪装成可编辑表单：

- 当前生效广告源。
- 原生端最低版本或兼容状态。
- 激励回调/奖励校验状态。
- 最近配置更新时间和操作人。

### 6.2 移出字段

从广告接入表单移除：

- 穿山甲账号名称。
- Taku 账号名称。
- 当前主链路未消费的 Taku App Secret。
- 当前 Taku-only 激励流程未使用的穿山甲解锁广告位。
- 报表专用 Publisher Key、时区、币种和金额精度。
- 仅由系统计算或返回的内部版本号、回调状态等只读值。

后端可以暂时兼容旧字段读取以支持平滑迁移，但新接口响应和审计不得回显密钥明文；确认无调用方后再单独做字段废弃。

### 6.3 必填与密钥语义

- 平台开关关闭时，该平台运行字段不要求填写。
- 首次启用或明确执行“更换密钥”时，关联运行字段必须完整填写。
- 已配置且后端只返回 `configured=true` 的密钥视为满足必填；编辑其他字段时留空表示保留原密钥。
- 密钥输入框永不回显原值，界面显示“已配置”状态和“更换”动作。
- 保存时仅发送用户新输入的密钥；不得用空字符串覆盖已配置密钥。

## 7. 收益报表

【收益报表】独立承载 Taku 官方收益报表配置：

- Publisher Key。
- 报表时区。
- 币种。
- 金额精度/缩放。
- 凭证是否已配置、凭证版本。
- 最近同步时间、最近成功时间、最近错误与权限诊断。

Publisher Key 采用与广告密钥相同的只写不读语义。报表配置只影响官方收益同步、核对、BI 和结算口径，不影响客户端广告请求；页面必须明确提示这一差异。

收益报表在代理商工作台内作为与【广告接入】同级的独立标题，不再作为【广告接入】内部小节。本期不新增平台级全局收益菜单：超级管理员先在【代理商管理】中选择任意动态代理商，代理商管理员则自动进入自己的租户工作台，两种角色复用同一收益报表页面和服务端权限规则。

## 8. 数据流与错误处理

### 8.1 数据流

1. 前端只提交筛选条件和允许修改的业务字段。
2. 服务端根据登录身份确定可访问租户范围。
3. 超级管理员选择的代理商 ID 仅作为合法的缩小范围条件。
4. 服务端完成查询、配置版本更新和审计记录。
5. 前端展示服务端返回的真实状态，不自行推断“已接入”“已同步”或“有收益”。

### 8.2 错误处理

- 401：清理失效会话并跳转登录页。
- 403：显示无权访问，不降级展示其他租户数据。
- 404：区分资源不存在与前端路由不存在；兼容路由只处理已知旧链接。
- 配置校验错误：精确定位到字段，保留用户已填写内容。
- 报表同步失败：展示服务端错误摘要、发生时间和重试条件，不把失败数据计入可结算收益。

## 9. 测试与验收

### 9.1 路由回归测试

- 单元测试覆盖绝对子路径、相对子路径、空路径和外部 URL。
- 菜单渲染测试断言广告中心两个菜单最终 href 分别为 `/skit/ad-consumption` 和 `/skit/ad-record`。
- 直接访问正式路径不显示 404。
- 访问两个旧错误路径会跳转到正式路径。
- 面包屑、标签页和 TabMenu 不再产生重复 `/skit`。

### 9.2 租户隔离测试

- 超级管理员不筛选时可查看全平台，筛选后只返回目标代理商。
- 代理商管理员即使手工传入其他 tenantId，也只能得到自己租户的数据或 403。
- 新增代理商无需前端改动即可进入筛选列表。
- 用户、广告记录、收益报表各自验证相同隔离规则。

### 9.3 配置测试

- 首次启用缺少运行字段时保存失败。
- 已配置密钥留空编辑其他字段时不会覆盖密钥。
- 显式更换密钥时完成新密钥校验和版本更新。
- 被移除的无效字段不再阻塞保存。
- 报表配置与广告运行配置分别保存、分别鉴权、分别审计。

### 9.4 发布验收

- 运行相关单元测试、类型检查和生产构建。
- 在真实角色下分别验证超级管理员与代理商管理员。
- 生产部署后用浏览器验证菜单点击、直接 URL、刷新和兼容跳转。
- 通过网络请求确认租户范围来自服务端，而不是仅依赖页面隐藏。

## 10. 非目标与后续项

本次不通过前端改造解决第三方广告库存不足、上游无出价或 Taku 后台广告源未配置问题；这些需要结合 SDK 日志和 Taku 平台诊断。此次改造保证管理侧能准确展示配置与真实状态，不把外部无填充误报为本地配置成功。

后续可在稳定的消费记录和官方收益数据上继续建设 BI，包括请求量、填充率、展示率、完成率、奖励成功率、eCPM、收益和异常漏斗，并确保每项指标口径可追溯到原始广告消费记录。
